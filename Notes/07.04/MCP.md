### MCP 的核心架构

MCP 采用标准的 Client–Server 架构，主要包含三个角色：

**1. Host**

运行模型和 Agent 的环境。

例如：

- Claude Desktop / Cursor
- 你自己的 Agent 平台
- 企业内部 AI 系统

**2. Client**

Host 内部的协议模块，负责：

- 发现 MCP Server
- 建立安全连接
- 管理上下文交互

它是 AI 世界的“网卡 + 驱动”。

**3. Server**

真正承载能力的独立服务。

例如：

- 文件系统 MCP Server
- 数据库 MCP Server
- GitHub MCP Server
- 内部系统 MCP Server

它们不属于某个 Agent，而是**系统能力节点**。

### MCP 的三大核心功能

通过 MCP，Server 可以向 AI 提供三种主要能力：

| 功能                     | 说明                                                         | 例子                                                |
| ------------------------ | ------------------------------------------------------------ | --------------------------------------------------- |
| **Resources (资源)**     | 允许 AI **读取**数据。类似于文件的只读访问。                 | 读取一份 `.md` 文档或数据库的内容。                 |
| **Tools (工具)**         | 允许 AI **执行**操作。这是我们熟悉的 Tool Calling 的标准化版。 | 在 GitHub 上创建一个 Issue，或发送一条 Slack 消息。 |
| **Prompts (提示词模板)** | 预设的对话指令。                                             | 提供一个“代码审查”或“撰写周报”的固定模板。          |

### MCP的工程价值

- **一次编写，到处运行**：如果你开发了一个接入自研数据库的 MCP Server，它不仅能在你的应用里用，还能直接被 Claude Desktop 或其他支持 MCP 的 IDE 识别。
- **隐私与安全**：数据源不再需要直接暴露给云端模型。MCP Server 运行在你的本地或私有环境中，只有经过处理的上下文会被发送给 AI。
- **生态互联**：目前已经有大量现成的 MCP Server（如访问本地文件、Google Search、PostgreSQL、GitHub 等），你可以像安装插件一样快速扩展 AI 的能力。



# 为什么MCP可以做到这样，底层逻辑是怎么实现的？
MCP 能做到 “**一次开发，所有 AI 都能用**”，**不是因为魔法，是因为它把「工具」和「AI」彻底拆开，再用统一协议连起来。**

------

## 1. 先搞懂：以前为什么不行？

以前的工具调用（Function Calling）：

- **模型 → 工具** 是**绑死**的

- GPT 有自己的格式

- Claude 有自己的格式

- LangChain 有自己的格式

  → 你给每个 AI 都要写一遍适配代码

就像：

- 苹果一个插头

- 华为一个插头

- 电脑一个插头

  互相不通用

------

## 2. MCP 的核心设计：三层解耦

MCP 把系统拆成 **3 个完全独立的东西**：

1. **MCP Server（工具提供者）**
   - 你写的工具：读文件、查数据库、调 API
   - 只负责**做事**
   - 不管谁来调用
2. **MCP Client（协议翻译官）**
   - 内置在 AI 里（GPT、Claude、VS Code、LangChain）
   - 只负责**说话**（按统一格式通信）
3. **Host（AI 应用本身）**
   - 决定要不要调用、权限给多少

它们之间**只通过统一协议说话**，

谁也不依赖谁。

------

## 3. 底层通信逻辑

MCP 用的是：**JSON-RPC 2.0**

你不用懂细节，只要记住：

##### 所有通信只有 3 种消息：

1. **你有什么能力？**

   Server 告诉 Client：

   - 我有这些工具
   - 我有这些文件资源
   - 我有这些提示词模板

2. **帮我执行这个工具**

   AI → Client → Server：

   - 工具名
   - 参数
   - 要求

3. **执行结果返回**

   Server → Client → AI：

   - 成功 / 失败
   - 结果内容

**全程格式统一，谁来都一样。**

------

## 4. 为什么 MCP 能跨模型、跨框架？

因为：

### **工具不绑定模型，模型不绑定工具。**

- 你写一个 MCP 工具 = 发布一个服务
- 任何 AI 只要装上 **MCP 客户端**，就能调用
- 工具不用改一行代码
- AI 也不用改内部逻辑

------

## 5. MCP 底层真正厉害的 3 点

##### ① 工具是独立进程

- MCP Server 是**单独运行**的程序
- 可以用 Python / JS / Go 随便写
- 可以跑在本地、服务器、Docker
- AI 只是**调用它**，不拥有它

##### ② 协议统一，不跟任何厂商走

- 不是 OpenAI 的协议
- 不是 Anthropic 的协议
- 是**公开、中立、标准**的协议

谁都能实现，谁都能用。

##### ③ 只定义 “怎么通信”，不定义 “怎么思考”

- MCP 不管 AI 怎么决策

- 不管 AI 用什么推理框架

- 只负责：

  把工具能力告诉 AI → AI 决定调用 → 执行 → 返回结果


## 总结
MCP 的底层逻辑是三层解耦架构 + 标准化通信协议：
1. 将工具封装为独立的 MCP Server，与模型、应用完全分离；
2. 使用 JSON-RPC 统一协议 进行通信，屏蔽不同模型的差异；
3. AI 通过 MCP Client 连接 Server，获取工具、资源、执行调用；
4. 实现一次开发、任意模型、任意应用都可调用，彻底解决工具生态碎片化问题.





# 问题

### 1. 什么是 MCP？

MCP = **Model Context Protocol（模型上下文协议）**，是一套**开放、标准化**的协议，用来统一 LLM 与外部工具、数据、服务的连接方式。



### 2. MCP 解决了什么问题？

- 各家模型 Function Calling 格式不统一
- 工具只能在单个框架 / 模型里用，无法复用
- 工具、应用、模型耦合太紧，难以维护
- 企业内部工具资产无法统一管理

一句话：**解决 AI 生态 “连接碎片化” 问题。**



### 3. MCP 三大核心组件是什么？

- **Host**：AI 应用 / IDE / 客户端（Claude、VS Code、LangChain）
- **Client**：协议客户端，负责通信
- **Server**：提供工具、资源、提示词的服务端



### 4. MCP 三大核心能力（Primitives）？

1. **Tools**：函数 / 工具调用
2. **Resources**：数据资源（文件、数据库）
3. **Prompts**：提示词模板

### 5. MCP vs 原生 Function Calling（OpenAI/Claude）？

- 原生：**厂商私有接口**，格式不统一，只能在对应模型用
- MCP：**开放标准协议**，一套格式通吃所有模型



### 6. MCP vs LangChain Tool Calling？

- LangChain Tool：**框架层封装**，只能在 LangChain 里用

- MCP：**跨框架、跨应用、跨语言**的通用标准

  关系：互补，LangChain 可以接入 MCP。



### 7. MCP 和插件系统（如 ChatGPT Plugins）区别？

- 插件：平台绑定、闭源、难复用
- MCP：**开放、本地可部署、通用、可热插拔**

### 8. MCP 通信流程？

1. 启动 MCP Server，暴露 Tools/Resources/Prompts
2. Host 通过 Client 连接 Server
3. 模型需要工具时，通过 MCP 发送调用请求
4. Server 执行并返回结果
5. 模型整合结果回答用户



### 9. 为什么说 MCP 是 “AI 的 USB‑C”？

- **一次开发，到处能用**
- 统一接口，不用为每个模型 / 应用写适配
- 热插拔，即插即用



### 10. MCP 为什么比传统工具调用更适合企业？

- 工具可**独立部署、独立维护**
- 权限统一管控
- 跨项目、跨应用复用
- 不绑定模型 / 框架

### 11. MCP 适合哪些场景？

- 企业内部工具统一接入 AI
- 多模型（GPT/Claude/ 国产）共用一套工具
- IDE 插件、本地 AI 应用
- 需要安全、隔离、可审计的工具调用



### 12. MCP 的优势？

1. **跨模型、跨框架、跨应用**
2. 解耦：应用与工具分离
3. 标准化，减少适配成本
4. 安全可控、可扩展
5. 支持本地 / 远程部署

### 13. Resources 和 Tools 的区别？

- **Resources**：数据 / 文件 / **只读**资源
- **Tools**：可执行的动作 / 函数



### 14. MCP 未来会取代 LangChain 吗？

不会。

- **MCP = 连接标准**

- **LangChain = 应用开发框架**

  未来趋势：

  LangChain 集成 MCP，MCP 为 LangChain 提供外部工具。

### 总结

MCP 是模型上下文协议，一套开放标准，用来统一大模型与外部工具、数据的连接。它解决了原生函数调用格式混乱、无法跨模型复用的问题。核心是 Tools、Resources、Prompts 三大能力，采用 Host-Client-Server 架构，真正做到一次开发、任意模型、任意应用都能调用，被称为 AI 时代的 USB‑C。



## 例子

### 例子 1：AI 想读你电脑里的文件

##### 不使用 MCP

- 你想让 Claude 看你电脑里的代码
- 你要：手动复制 → 粘贴给 AI
- 很麻烦，不能直接读

##### 使用 MCP

你在电脑上**启动一个 MCP 服务**：

```
mcp-server-files
```

这个服务告诉 AI：

- 我能读文件
- 我能写文件
- 我能列出文件夹

然后：

- Claude（AI）**直接通过 MCP 读你电脑文件**
- 不用你复制粘贴
- 安全、可控、不用传数据给别人

这就是 **MCP**。

------

### 例子 2：AI 想查你公司数据库

##### 不使用 MCP

- GPT 不能连你公司数据库
- 你要自己写代码、写接口、做适配

##### 使用 MCP

你启动一个：

```
mcp-server-database
```

它告诉所有 AI：

- 我能查数据库
- 我能执行 SQL
- 我有权限控制

然后：

- **GPT 能用**
- **Claude 能用**
- **LangChain Agent 也能用**

不用给每个模型写一套代码，**一次写，全部 AI 都能用**。

这就是 MCP。

------

### 例子 3：你有一个天气工具，想给所有 AI 用

##### 不使用 MCP

- 给 GPT 写一套函数调用格式

- 给 Claude 再写一套

- 给豆包再写一套

  累死

##### 使用 MCP

你写一个：

```
mcp-server-weather
```

然后：

**所有支持 MCP 的 AI，都能直接用这个工具**

不用改代码、不用适配、不用重新开发。
