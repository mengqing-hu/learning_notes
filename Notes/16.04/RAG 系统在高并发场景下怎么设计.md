# RAG 系统在高并发场景下怎么设计？

### 一、高并发 RAG 系统核心设计思路

**核心目标：低延迟、高吞吐、不雪崩、可水平扩展。**

整体思路：**缓存前置 + 异步削峰 + 无状态服务 + 分布式检索 + 资源隔离**。

### 二、高并发架构设计（分层讲）

#### 1. 接入层

- 使用 **API 网关**（Kong/Spring Cloud Gateway）
- 做 **全局限流、熔断、黑白名单、鉴权**
- 防止恶意流量打垮后端

#### 2. 应用服务层（LLM + RAG 逻辑）

- 服务 **无状态化**，可横向扩容（K8s 部署）
- 接口 **批量推理 + 异步化**
- 热点问题 **多级缓存**
  - L1：本地内存缓存（Caffeine）
  - L2：Redis 分布式缓存（问题 → 检索结果 / 直接答案）
- 针对相同问题直接返回缓存，**不再走检索 + LLM**

#### 3. 检索层（高并发最关键）

##### （1）向量检索优化

- 使用 **Milvus/ES 集群**，分布式分片部署
- 向量索引使用 **HNSW**（低延迟）
- 关闭实时插入，改用 **近实时（Near-Real-Time）** 批量导入
- 召回数量控制：**top 20～40**，不做大批量召回

##### （2）精排层（Cross-Encoder）高并发方案

Cross-Encoder 慢，高并发下必须优化：

- **模型小尺寸化**：用 base 版，不用 large
- **批量推理**（batch 处理）
- **GPU 推理加速**（TensorRT/VLLM/Triton）
- 精排前先用 **RRF 快速过滤**，减少输入 Cross-Encoder 的数量
- 精排结果 **缓存**，相同问题不再重排

#### 4. LLM 推理层

- 使用 **vLLM / TGI 推理引擎**，高吞吐
- 开启 **动态批处理、PagedAttention**
- LLM 前面加 **缓存层**（Redis），热点问题直接返回
- 做 **请求排队、限流、优先级队列**

#### 5. 存储层

- 向量库：Milvus/ES 分布式集群
- 文档库：MySQL/PG 分库分表
- 缓存：Redis 集群

#### 6. 流量控制（高并发保命三件套）

- **限流**：单机限流 + 网关全局限流
- **熔断**：下游超时自动熔断，防止雪崩
- **降级**：并发过高时，关闭精排、只走向量检索，保证可用

## 三、高并发 RAG 请求完整流程

1. 用户请求 → 网关限流鉴权
2. 查 Redis 缓存，命中直接返回
3. 未命中 → 进入 RAG 流程
4. BM25 + 向量检索双路召回（分布式并行）
5. RRF 融合 → 得到候选集
6. Cross-Encoder 批量精排（GPU 加速）
7. 构造 Prompt → 送入 LLM 批量推理
8. 结果写入 Redis 缓存
9. 返回给用户

## 四、关键优化点

1. **缓存优先**：热点问题占比通常很高，缓存能扛 70%+ 流量
2. **检索并行化**：BM25 和向量检索同时发请求，降低总耗时
3. **精排轻量化**：Cross-Encoder 只处理少量候选，避免瓶颈
4. **无状态服务**：随便扩容副本
5. **削峰异步**：超高并发可转异步任务，避免阻塞
6. **资源隔离**：检索、精排、LLM 独立部署，互不影响
7. **近实时更新**：知识库不实时写入，降低向量库压力

## 五、总结

高并发 RAG 系统通过**多级缓存、分布式检索、服务无状态化、GPU 加速精排与 LLM 推理、限流熔断降级**，实现低延迟、高吞吐、水平扩展；核心是**减少重复检索与精排，让缓存扛流量，让并行提速度，让削峰保稳定**。
