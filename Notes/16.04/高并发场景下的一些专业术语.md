# 1. 低延迟

- 人话：反应快，不墨迹，用户发请求后，能快速拿到结果。
- 例子：点外卖下单后，秒出订单确认；刷短视频，一点就加载播放，不转圈圈。
- 技术方案：用 HNSW 向量索引（检索快）；接口用 HTTP/2 协议；减少多层调用，核心逻辑直接执行；本地缓存优先查询。

# 2. 高吞吐

- 人话：一次性能扛住很多人同时用，不会因为用户多就卡壳、崩溃。
- 例子：奶茶店高峰时，一秒能出 10 杯奶茶，同时接待 20 个顾客，不耽误事；抖音双 11，百万用户同时刷视频也不卡顿。
- 技术方案：服务无状态化，支持多副本部署；接口批量处理请求；用 GPU 加速模型推理（vLLM/TGI）；向量库分布式分片。

# 3. 不雪崩

- 人话：一个地方炸了（比如某个模块故障），别连累整个系统，保证系统整体能正常用。
- 例子：食堂一个打饭窗口阿姨请假，其他窗口照样营业，不会整个食堂停摆；手机相机坏了，不影响打电话、发微信。
- 技术方案：给每个服务加熔断（Sentinel/Resilience4j）；超时设置（防止无限等待）；降级策略（故障时关闭非核心功能）；服务之间隔离调用。

# 4. 可水平扩展

- 人话：人多了、请求多了，就直接加机器，不用重构整个系统，加完就能扛更多压力。
- 例子：奶茶店高峰时，原本 2 个窗口不够，再临时加 3 个窗口，不用重新装修店面；超市人多，再开 2 个收银台。
- 技术方案：用 K8s 部署服务，一键增加副本数；向量库 / 数据库分片扩容（Milvus/ES 新增节点）；所有服务无状态，新增机器直接接入。

# 5. 缓存前置

- 人话：热门答案提前存好，用户问的时候直接给，不用每次都重新查资料、重新计算。
- 例子：老师把学生最常问的 10 个问题（比如 “作业是什么”），提前写在黑板上，有人问直接指黑板，不用反复回答。
- 技术方案：热门请求结果存到 Redis（分布式缓存）和本地内存（Caffeine）；缓存 key 用问题原文 / 哈希值，value 存最终回答；设置合理过期时间（比如 1 小时）。

# 6. 异步削峰

- 人话：高峰期请求太多，不让所有请求挤在一起冲系统，先放进队列，后台慢慢处理，避免系统被冲垮。
- 例子：春运火车站，人太多不允许一窝蜂进站，先让大家在广场排队，按顺序分批进站；外卖高峰，订单先排队，骑手按顺序取单配送。
- 技术方案：用消息队列（RabbitMQ/Kafka/Redis List）接收请求；后台 worker 线程慢慢消费队列；前端显示 “排队中”，处理完通过 WebSocket 通知结果。

# 7. 无状态服务

- 人话：来一个请求，哪个机器闲就哪个处理，每个服务节点不记用户信息，换机器也不影响请求处理。
- 例子：食堂打饭，哪个窗口阿姨有空就去哪个，不用固定找某一个；小区快递柜，哪个有空就存哪个，柜子不记是谁的快递（只记取件码）。
- 技术方案：服务不存储用户会话、请求历史，所有用户信息存在 Redis / 数据库；请求携带 token / 会话 ID，每个节点都能通过 ID 查询用户信息；K8s 自动调度空闲节点处理请求。

# 8. 分布式检索

- 人话：把所有资料切成好几份，多台机器同时查自己负责的那部分，最后把所有机器的查询结果汇总，快速找到最相关的内容。
- 例子：10 个图书管理员一起帮你找一本书，每个人负责一个区域，找到后汇总，比一个管理员找快 10 倍；快递分片区配送，多个快递员同时送，效率翻倍。
- 技术方案：向量库（Milvus/ES）做分片，每台节点存 1/N 数据；查询时，多线程同时请求所有分片；汇总所有分片结果，按相关性排序；用 ThreadPoolExecutor 实现并行查询。

# 9. 资源隔离

- 人话：各个模块各干各的，互不干扰，一个模块炸了、忙了，不影响其他模块正常工作。
- 例子：厨房分炒菜区、炸物区、凉菜区，炒菜区着火了，炸物区、凉菜区还能正常工作；手机微信卡了，不影响电话、短信功能。
- 技术方案：检索、精排、LLM 独立部署，用不同的服务器 / GPU；用容器（Docker）隔离资源；每个模块设置独立的内存、CPU 配额，互不抢占。

# 10. 全局限流

- 人话：系统有最大承载能力（比如最多扛 1000 人同时访问），来了 2000 人，只放 1000 人进来，剩下的直接拦在外面，或者让等一会儿，防止系统被挤爆。
- 例子：网红店一次只放 20 人进店，多了就在门口排队；游乐园过山车，一次只坐 8 人，超过的人排队等候。
- 技术方案：用 API 网关（Kong/Spring Cloud Gateway）做全局限流；用令牌桶 / 漏桶算法控制请求数；超过限流阈值时，返回 “请稍后再试”，或加入排队队列。

# 11. 熔断

- 人话：某个服务坏了、反应太慢了，就直接暂时关掉它，不让请求再往这个服务发，别连累整个系统卡死。
- 例子：电梯超重了，直接熔断关门，不运行，防止电梯故障；电器短路，保险丝熔断，保护整个电路不被烧坏。
- 技术方案：用 Sentinel/Resilience4j 实现熔断；检测服务响应时间（比如超过 3 秒）或失败率（比如 50% 请求失败），自动触发熔断；熔断后，请求直接返回降级结果，一段时间后尝试恢复。

# 12. 白名单

- 人话：只允许指定的人 / IP 进入系统、调用接口，其他人进不来。
- 例子：公司大门，只有员工能刷工牌进入，外人不让进；小区门禁，只有业主在白名单里，能刷卡开门。
- 技术方案：API 网关配置白名单（公司内部 IP、信任的第三方 IP）；用户层面，只允许指定账号（管理员、内部用户）访问核心接口；用 Nginx 配置白名单，直接拦截非白名单请求。

# 13. 黑名单

- 人话：指定的人 / IP 永远不准进入系统、调用接口，比如恶意攻击、刷接口的对象。
- 例子：网吧禁止未成年人进入，未成年人就在黑名单里；小区禁止小偷、闹事者进入，加入黑名单。
- 技术方案：API 网关配置黑名单（恶意 IP、频繁刷接口的 IP）；用户层面，封禁违规账号（恶意注册、攻击系统的账号）；用防火墙拦截黑名单 IP，禁止其访问服务器。

# 14. 鉴权

- 人话：验证你是谁，有没有资格访问系统、调用接口，防止陌生人乱搞数据，不是谁都能随便用。
- 例子：进公司要刷工牌（验证身份），没工牌不让进；登录微信要输密码 / 验证码，验证是本人才能登录。
- 技术方案：用 JWT token、OAuth2.0 做身份校验；登录时验证账号密码，通过后发放 token；接口请求时，携带 token 验证，无 token / 无效 token 直接拒绝；核心接口额外加权限校验（比如只有管理员能访问）。

# 15. 接口批量推理

- 人话：把多个用户的问题攒成一组，一起送给模型计算，而不是一个一个算，提高效率、减少等待。
- 例子：快递员不一个包裹送一趟，而是一车拉几十个包裹一起送，节省时间；老师批改作业，攒一批一起批改，比一本一本批改快。
- 技术方案：接口支持接收多个 query 一次性传入；用 vLLM/TGI 推理引擎开启动态批处理；服务端用队列攒批（达到 10 个请求或超时 50ms，就一起推理）；模型一次 forward 处理多条数据，提升吞吐量。

# 16. 异步化

- 人话：用户发请求后，不用站在原地死等结果，能去干别的，后台慢慢计算，算完再通知用户。
- 例子：去餐厅点餐，不用站在厨房门口等，坐下玩手机，菜做好了服务员喊你；寄快递，填完单就可以走，快递员后续慢慢处理配送。
- 技术方案：用 FastAPI async、Tornado 等异步框架开发接口；请求进入消息队列，后台 worker 异步消费处理；结果通过 WebSocket 或轮询接口返回给前端；避免同步等待导致线程耗尽、请求超时。

# 17. 热点问题多级缓存

- 人话：大家经常问的热门问题，把答案提前存起来，分多层存储（越近越快），下次有人问，直接返回答案，不再走检索和 LLM 计算。
- 例子：老师把最常问的 100 道题答案，写在口袋小纸条（最快）、讲台抽屉（次快）、办公室文件柜（备用），有人问直接掏，不用翻书。
- 技术方案：L1 缓存（本地内存，Caffeine / 内存字典），速度最快，存最热门的请求；L2 缓存（Redis 分布式缓存），全局共享，存所有热门请求；缓存 key 用问题原文 / 哈希值，value 存最终回答；定时过期（比如 1 小时），知识库更新时主动删除对应缓存。


# 18. 数据库分库分表

- **人话**：数据太多一张表放不下、查太慢，就把数据拆成很多库、很多表，分开存、分开查。
- **例子**：一个书架放不下所有书，就分 10 个书架，按类别 / 年级分开放。
- **技术方案**：按用户 ID 哈希分片、按时间分表；用 Sharding-JDBC/MyCat 做分片；查询时路由到对应库表。

# 19. 读写分离

- **人话**：查数据（读）和改数据（写）分开用不同数据库，不互相抢资源。
- **例子**：前台收银只管结账（写），后台查账只管看报表（读）。
- **技术方案**：一主多从，主库写、从库读；MySQL 主从复制；业务自动路由读写。

# 20. 接口幂等性设计、防重复提交

- **人话**：同样的请求点多次，效果只算一次，不会重复扣款、重复创建订单。
- **例子**：转账点两次，只转一次钱；外卖提交两次，只下一单。
- **技术方案**：前端按钮置灰；后端用唯一订单号 / Token；数据库唯一索引；Redis 分布式锁防重。

# 21. 流量削峰、队列缓冲（MQ 异步解耦）

- **人话**：突然一大波请求涌进来，先放进队列排队，系统慢慢处理，不被冲崩。
- **例子**：早高峰地铁限流，人先在排队区等，分批放进站。
- **技术方案**：用 RabbitMQ/Kafka/Redis 队列；请求先进队列，后台异步消费；削峰填谷。

# 22. 服务注册发现、配置中心统一管理

- **人话**：很多服务互相调用，要知道对方在哪；配置统一改，不用每个服务都改一遍。
- **例子**：公司各部门电话簿统一更新；规章制度统一发，不各自打印。
- **技术方案**：Nacos/Eureka/Consul 注册服务；Nacos/Apollo 做配置中心；自动下发配置。

# 23. 网关统一入口、鉴权、路由、限流

- **人话**：所有请求先进一个大门，统一验身份、指路、控制人数，不让乱进。
- **例子**：小区大门保安，查门禁、指路、人多限流。
- **技术方案**：Spring Cloud Gateway/Kong；统一鉴权、路由、限流、日志、跨域。

# 24. 分布式锁、防止并发冲突

- **人话**：多个人同时改同一份数据，只允许一个人先改，避免改乱。
- **例子**：厕所一次只能进一个人，上锁排队。
- **技术方案**：Redis 分布式锁；Redlock；ZooKeeper 锁；防止超卖、重复扣款。

# 25. 数据冗余备份、主从高可用

- **人话**：数据多存几份，一台机器挂了，另一台立刻顶上，不丢数据、不停服务。
- **例子**：重要文件 U 盘 + 云盘各存一份；手机坏了换手机还在。
- **技术方案**：MySQL 主从；定期备份；binlog 同步；自动主从切换。

# 26. 异地多活、容灾切换

- **人话**：一个城市机房炸了，另一个城市机房立刻顶上，用户无感知。
- **例子**：一家店关门，另一家分店立刻接单。
- **技术方案**：多机房部署；流量 DNS / 网关切换；数据跨城同步。

# 27. 超时重试、熔断重试、优雅降级

- **人话**：调用别人服务暂时失败，就重试几次；实在不行就跳过，保证主功能能用。
- **例子**：打电话没人接，多拨两次；实在打不通就先发短信。
- **技术方案**：设置合理超时；重试次数限制；Sentinel/Resilience4j 熔断；降级返回默认值。

# 28. 静态资源 CDN 加速

- **人话**：图片、JS、CSS 放全国节点，用户就近取，加载飞快。
- **例子**：快递在全国建仓，你附近发货，更快。
- **技术方案**：阿里云 / 腾讯云 CDN；缓存静态资源；就近访问。

# 29. 防刷、防爬虫、接口鉴权与风控

- **人话**：防止恶意频繁请求、脚本刷接口，保护系统和数据。
- **例子**：网红店限每人限购，防止黄牛批量扫货。
- **技术方案**：IP 限流；验证码；设备指纹；行为风控；签名验签；JWT 鉴权。

# 30. 链路追踪、日志聚合、问题快速定位

- **人话**：一个请求经过很多服务，全程记录，出问题一眼看到哪一步挂了。
- **例子**：快递物流轨迹，到哪了一清二楚。
- **技术方案**：SkyWalking/Pinpoint/ELK；全链路 ID；日志集中收集。

# 31. 资源隔离（线程池、集群、机房隔离）

- **人话**：不同业务各用各的资源，一个崩了不连累别人。
- **例子**：厨房分炒菜、油炸、凉菜区，互不影响。
- **技术方案**：独立线程池；独立集群；机房级隔离；Docker/K8s 资源限制。

# 32. 预热缓存、缓存预热、防止缓存击穿

- **人话**：系统启动前 / 高峰期前，提前把热门数据塞进缓存，不让请求直接打库。
- **例子**：冬天提前开暖气，回家就暖和。
- **技术方案**：启动时加载热点数据；定时预热；互斥锁构建缓存。

# 33. 布隆过滤器、防止缓存穿透

- **人话**：过滤掉根本不存在的查询，不让它们每次都去查数据库。
- **例子**：门卫先查名单，不在名单的直接拒绝，不用层层上报。
- **技术方案**：Redis 布隆过滤器；白名单过滤不存在 ID；拦截无效查询。

# 34. 服务编排、服务治理

- **人话**：多个服务按顺序调用，统一管理超时、重试、异常。
- **例子**：办业务按步骤：取号→填表→审核→领证。
- **技术方案**：Spring Cloud Stream/Seata；统一监控、限流、降级、链路管理。

# 35. 弹性伸缩、自动扩缩容

- **人话**：人多自动加机器，人少自动减机器，省钱又不崩。
- **例子**：夏天空调自动调风速，热就大风，凉就小风。
- **技术方案**：K8s HPA；根据 CPU / 流量自动扩缩副本；云上弹性 ECS。

# 36. 混沌工程、故障演练

- **人话**：主动制造故障（杀进程、断网），看系统会不会崩，提前补强。
- **例子**：消防演练，模拟着火看大家会不会逃。
- **技术方案**：Chaos Mesh；随机杀 Pod、网络延迟、数据库宕机演练。

# 37. 数据分片、冷热数据分离

- **人话**：经常用的热数据放快盘，不常用的冷数据放便宜慢盘。
- **例子**：常用书放桌面，不常用的放储藏室。
- **技术方案**：按时间分表；热数据放 SSD/Redis；冷数据归档到对象存储 / OSS。
