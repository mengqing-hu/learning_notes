# 文档处理与数据加载



## 多格式解析：先把“文件”变成“正确的文本”

### 按格式选工具

不同的格式需要用不同的工具来读取，整理如下:

| 格式                            | 场景                 | 推荐思路 / 工具                                  | 说明                                                         |
| ------------------------------- | -------------------- | ------------------------------------------------ | ------------------------------------------------------------ |
| **PDF**（排版复杂、两栏、含表） | 报告、论文、手册     | **mineru**、**marker**                           | 做版面分析 + 阅读顺序恢复 + 表格结构还原，而非简单按字符流提取 |
| **PDF**（纯文字、单栏）         | 简单文档             | pypdf / pdfplumber                               | 可直接提取，但仍建议保留页码等元数据                         |
| **Image**（合同拍照、截图）     | 扫描件、截图         | **OCR**（如 PaddleOCR、Tesseract）+ 版面分析     | 先 OCR 再按块排序；复杂版式可配合 marker 等                  |
| **Excel**                       | 表格、清单、数据说明 | **pandas**（read_excel）、**openpyxl**           | 按 sheet/行列读取，表格转 Markdown 或结构化文本，便于后续分块 |
| **Word**                        | 合同、说明文档       | python-docx、**LangChain UnstructuredLoader**    | 保留段落与标题层级，便于做语义分块                           |
| **网页 / Markdown**             | 在线文档、笔记       | BeautifulSoup、markdown 解析、**Readability** 等 | 去掉导航、广告，只保留正文结构                               |

**mineru**、**marker** 这类工具的价值在于：不只做 OCR 或文字提取，而是做**版面块识别**（标题 / 正文 / 表格 / 图片）、**阅读顺序恢复**、**表格结构还原**，输出更接近“人类阅读顺序”的文本，适合作为 RAG 的输入。

### 使用时的注意点

- **PDF**：优先区分“可选中文字”与“扫描图”。可选中文字可用版面分析工具（mineru、marker）直接解析；纯扫描件必须先 OCR 再按块排序。
- **Image**：单张图片可先 OCR 得到文本，再进入 ETL；若多页或与 PDF 混合，建议先转成统一结构（如每页一段 + 页码 metadata）。
- **Excel**：按 sheet 或按“表头+行”组织成一段段说明性文本，并带上 `sheet_name`、`source` 等 metadata，方便后续按来源过滤。





## 数据清洗（ETL）规范：提升 Embedding 的信号质量

解析得到的文本里，往往还夹杂着**多余空格、换行、页眉页脚、乱码、OCR 错字**等。如果不清洗，这些都会被一起 Embedding，占用 token、稀释语义，检索时引入噪音。

Embedding 模型不会区分“有用”和“垃圾”，它只会**忠实编码一切**。所以需要一套固定的 **ETL 规范**，在进向量库前统一处理。

### 1. 常见“脏”文本长什么样

```
This   is     a   sentence.

———  PAGE  12  ———

§￥%……&*（乱码）
0 与 O、1 与 l 混淆（OCR 错误）
```

不清洗就进库，这些都会被视为“有意义的信息”，影响后续分块和检索质量。

### 2. 必做清洗项

| 类别               | 做法                                                 | 目的                                 |
| ------------------ | ---------------------------------------------------- | ------------------------------------ |
| **空白**           | 合并连续空格、制表符、换行为单一空格或按段落保留换行 | 减少无意义 token，保持可读性         |
| **页眉页脚**       | 去掉 “PAGE 12”、 “———”、 页码装饰线等                | 避免每页重复噪音进入检索             |
| **乱码与特殊符号** | 用正则去掉或替换 §￥%…&* 等无意义字符                | 提高 Embedding 信号密度              |
| **标点与编码**     | 全角/半角统一、编码统一为 UTF-8                      | 避免同一内容因写法不同被拆成两套向量 |
| **表格**           | 转成 Markdown 或“表头 + 行”的线性文本                | 便于按语义分块，而不是被拆成碎片     |
| **OCR 错误**       | 规则或小模型纠正 0/O、1/l、常见错别字                | 提升专有名词、数字的检索命中率       |

**核心目标不是美观，而是：让每个 token 尽量承载语义，而不是噪音。** 这一步会直接影响到后面的检索准确率；很多 RAG 效果差，不是模型问题，而是 **ETL 根本没做**。

### 3. 清洗流水线顺序建议

建议按固定顺序执行，避免重复处理或互相干扰：

1. **编码统一**：确保全文 UTF-8，全角/半角按需统一。
2. **去掉页眉页脚、装饰线**：用**正则或规则匹配**常见模式并删除。
3. **合并空白**：多空格/换行合并，段落边界可保留一个换行。
4. **去除乱码与无意义字符**：保留字母、数字、常用标点、中文等，其余视业务决定保留或删。
5. **表格转结构化文本**：若前面 Loader 已输出表格结构，这里可再统一成 Markdown 或键值式文本。
6. **OCR 纠错**（可选）：对已知易错字符做替换或小模型纠错。



简单示例：

```
import re

def clean_text(text: str) -> str:
    if not text or not text.strip():
        return ""
    # 合并连续空白为单个空格
    text = re.sub(r"\s+", " ", text)
    # 去掉常见页眉页脚模式（按实际文档调整）
    text = re.sub(r"———\s*PAGE\s*\d+\s*———", "", text, flags=re.IGNORECASE)
    text = re.sub(r"第\s*\d+\s*页", "", text)
    # 去掉部分乱码字符（保留中文、英文、数字、常用标点）
    text = re.sub(r"[§￥%……&*]+", " ", text)
    return text.strip()
```





## 元数据管理

在加载时保留关键信息，为精准过滤做准备。

大多数 Loader 默认只返回“正文”，例如：

```
Document(page_content="...")
```

但在 RAG 里，我们还需要**页码、来源文件、作者、日期、章节**等信息，用于检索时的**过滤**和**溯源**。这些信息应在**加载与解析阶段就写入**，而不是事后补。

### 1. 理想形态：Document = 内容 + Metadata

目标结构应类似：

```
Document(
    page_content="（该块正文）",
    metadata={
        "page": 12,
        "source": "policy_2024.pdf",
        "author": "合规部",
        "date": "2024-01-01",
        "section": "Risk Control",
    }
)
```

这样在写入向量库时，每条 chunk 都带上**来源、页码、章节、日期**等，检索时就可以“**Metadata Filter + Vector Search**”：先按条件筛（如只看 2024 年、某作者、某章节），再在候选集里做向量检索，精度和可解释性都会上一个台阶。

### 2. 在加载时如何保留 Metadata

不同格式在解析时能拿到的信息不同，可以在 Loader 或解析后逻辑里统一挂到 `metadata`：

| 来源      | 可保留的典型字段                 | 说明                                                     |
| --------- | -------------------------------- | -------------------------------------------------------- |
| **PDF**   | `page`、`source`、总页数         | 用 pypdf/mineru/marker 等取页码；文件名或路径作 `source` |
| **Excel** | `sheet_name`、`source`、行列范围 | 读每个 sheet 时记录 sheet 名；文件名作 `source`          |
| **Word**  | `source`、标题层级、段落序号     | 用 python-docx 读段落时可带标题级别                      |
| **网页**  | `url`、`title`、抓取时间         | 爬虫或 Readability 解析时记录                            |
| **通用**  | `author`、`date`、`version`      | 从文件属性、文件名或文档内部解析（如 PDF 的 XMP）        |

只要在**构建 Document 的那一步**把能拿到的信息都塞进 `metadata`，后续分块、向量化、检索都能沿用，无需再回溯原始文件。

### 3. 检索时怎么用：为精准过滤做准备

知识库变大后，常见需求会变成：

- 只检索 **2024 年之后** 的文档
- 只检索 **某个作者** 或某份报告
- 只检索 **第 5 ～ 10 页** 或 **某一章节**
- 同一问题在不同**版本/部门**文档间区分

若没有 metadata，只能做“全库向量搜索”；有了 metadata，向量库（如 Milvus、Pinecone、LangChain 封装）一般都支持 **filter**：先按 `metadata` 条件过滤，再在结果里做相似度排序。这样既减少无关文档干扰，又便于回答时**引用出处**（如“见 policy_2024.pdf 第 12 页”），提升可信度。

## 总结

文档处理和预加载不是在“读文件”，而是在建知识地基。

| 阶段         | 目标                                                         |
| ------------ | ------------------------------------------------------------ |
| **Loaders**  | 按格式选工具（如 mineru、marker 处理复杂 PDF，OCR 处理图片，pandas/openpyxl 处理 Excel），把复杂格式还原为**正确语义顺序**的文本 |
| **ETL**      | 用固定流水线去掉乱码、多余空白、页眉页脚等，**提高 Embedding 信号密度**，减少噪音 token |
| **Metadata** | 在加载时保留**页码、来源、作者、日期、章节**等，为后续**精准过滤和溯源**做准备 |

当这三步都做对之后：

- 分块（Chunking）更容易切出语义完整的块
- Embedding 效果更稳定
- 检索更准、幻觉更少
