# B + 树 和 红黑树 对比

- **B + 树**：**多叉平衡树**，专为**磁盘、数据库索引**设计（MySQL 索引）
- **红黑树**：**二叉平衡树**，专为**内存、快速查找**设计（Java HashMap、TreeMap）

# 红黑树（Red-Black Tree）

## 1. 本质

- **自平衡二叉查找树**
- 每个节点要么红要么黑，通过颜色规则维持平衡
- 最长路径 ≤ 2× 最短路径，保证不会退化成链表

## 2. 特点

- 二叉结构：**一个父节点最多 2 个子节点**
- 每个节点都存 **key + value**
- 高度接近 `log₂n`，查询稳定 `O(logn)`
- **插入、删除、旋转**效率很高
- **只适合在内存中使用**

## 3. 典型用途

- Java：**TreeMap、TreeSet、HashMap 冲突链表转红黑树**
- C++ STL：map、set
- 操作系统：进程调度、虚拟内存管理



# B+ 树（B+ Tree）

## 1. 本质

- **多路平衡查找树**（一个节点能有很多子节点）
- MySQL InnoDB 索引标准结构

## 2. 核心特点

- ==**多叉**：一个节点能存几十个～上千个 key==
- ==**只有叶子节点存完整数据**，内部节点只存索引 key==
- **叶子节点用==双向链表==串起来**，范围查询极快
- 所有叶子在同一层，查询稳定 `O(logₘn)`

## 3. 为什么适合数据库 / 磁盘？

- 磁盘 IO 极慢，**减少 IO 次数 = 提升性能**
- B+ 树**矮胖**，高度通常 3～4 层就能存千万数据
- 二叉树会变得很高，IO 爆炸

# B + 树 vs 红黑树（最强对比）

|  对比项  |             B+ 树             |            红黑树             |
| :------: | :---------------------------: | :---------------------------: |
|   结构   |        **多叉平衡树**         |        **二叉平衡树**         |
| 节点存储 | 内部节点只存 key，叶子存数据  |    每个节点都存 key+value     |
| 叶子节点 |       连成**有序链表**        |            无链表             |
|   高度   |          矮胖，IO 少          |       高瘦，内存内极快        |
| 范围查询 |       极快（链表遍历）        |        慢，要反复回溯         |
| 适用场景 | **磁盘、数据库索引（MySQL）** | **内存、快速查找（TreeMap）** |
|   效率   |      磁盘场景碾压红黑树       |         内存场景极快          |

# MySQL 为什么用 B+ 树？

核心目标：**减少磁盘 IO、范围查询快、有序、稳定**

对比一下为什么不用别的：

- 二叉树：深度太深 → IO 多 → 慢
- B 树：节点存数据 → 每个节点存得少 → 树高 → IO 多
- **B+ 树：只有叶子节点存数据，内部节点只存键 → 节点更 “胖” → 树更矮 → IO 更少**

# B+ 树核心特点

1. **==只有==叶子节点存完整数据**

   内部节点只存索引键，不存真实行数据。

2. **叶子节点形成==有序链表==**

   支持**范围查询、排序、遍历**极快。

3. **所有叶子节点在同一层**

   查询效率稳定，都是 ==**O(log n)**== 次 IO。

4. **非叶子节点起到==索引==作用**

   让树更矮，磁盘 IO 大幅减少。
