# **把表格存进向量数据库**的不同方法

### 举例表格

|  国家  | 手续费 | 单笔限额  |
| :----: | :----: | :-------: |
|  美国  |  1.5%  | 5000 USD  |
| 新加坡 |  1.0%  | 10000 USD |
|  德国  |  1.2%  | 8000 EUR  |
|  英国  |  1.3%  | 7000 GBP  |

下面 3 种方式，就是**把这张表格存进向量数据库**的不同方法。

------

### 方式 1：整表向量化（把整张表当 1 条文本存）

##### 存储方式

把**整个表格**转成一段文本，生成**一个向量**。

##### 转成的文本（存入向量库）

```
美国手续费1.5%，限额5000 USD
新加坡手续费1.0%，限额10000 USD
德国手续费1.2%，限额8000 EUR
英国手续费1.3%，限额7000 GBP
```

##### 向量数据库里存什么？

- vector：整段文本生成的向量（1 个）
- doc_id：101
- content：上面那段长文本
- 元数据：表名、国家列表

##### 优点

- 适合查**整体问题**，比如：

  “各国手续费分别是多少？”

- 只存 1 个向量，占用空间小

##### 缺点

- 查**单个国家**不准
- 不能单独更新某一行

### 方式 2：逐行向量化（表格每一行 = 1 条向量）【最常用、最好用】

##### 存储方式

**一行一个向量**，把表格拆成 4 条独立数据。

##### 转成的文本（4 条）

```
1. 美国，手续费1.5%，单笔限额5000 USD
2. 新加坡，手续费1.0%，单笔限额10000 USD
3. 德国，手续费1.2%，单笔限额8000 EUR
4. 英国，手续费1.3%，单笔限额7000 GBP
```

##### 向量数据库里存什么？

存 **4 个向量**，每个向量对应一行：

- id=1, vector = 向量 1, country = 美国，fee=1.5%
- id=2, vector = 向量 2, country = 新加坡，fee=1.0%
- ...

##### 优点（RAG 首选！）

- 查 “德国手续费” 能**精准命中德国那一行**
- 可以单独过滤国家 `country="DE"`
- 更新某一行不影响其他行

##### 缺点

- 向量数量变多

### 方式 3：表格结构化元数据 + 向量（字段存 metadata）

##### 存储方式

不存长文本，只存：

- 向量
- 结构化字段（country、fee、limit）全部存在 metadata 里

##### 向量数据库里存的样子（JSON）

```
id: 1
vector: xxx
metadata: {
  "country": "美国",
  "fee": "1.5%",
  "limit": "5000 USD"
}
```

```
id:2
vector:xxx
metadata: {
  "country": "新加坡",
  "fee": "1.0%",
  "limit": 10000
}
```

##### 优点

- 可以做**过滤查询**

  例如：查限额 > 5000 的国家

- 纯结构化，LLM 不容易理解错

##### 缺点

- 语义检索效果不如纯文本好

------

### 精简总结

我用最简单的话帮你总结：

1. **整表向量化**

   整张表格 → 一段文本 → 1 个向量

   → 适合查整体问题

2. **逐行向量化（最常用）**

   表格每一行 → 一段文本 → 1 个向量

   → 精准、可过滤、支付场景首选

3. **元数据存储**

   不存文本，只存字段（country/fee/limit）+ 向量

   → 适合做结构化过滤

### 在RAG 系统中，表格怎么存向量库？

在RAG 系统中，表格类数据不会直接存储，而是根据业务需求选择三种方式：

1. **整表向量化**，将整个表格转为单一文本向量，适合整体查询；
2. **逐行向量化**，每一行生成独立向量，这是我们最常用的方式，检索精准、支持过滤；
3. **结构化元数据 + 向量**，将表格字段以 JSON 格式存在元数据中，方便数值过滤。

实际业务中，**支付手续费、汇率、限额这类表格，我们都使用逐行向量化**，效果最好。
