# RAG 两阶段检索流程

RAG 的两阶段检索，核心是**先粗筛不漏、后精排不错**：

1. **粗筛阶段**：用 BM25 关键词检索 + 向量语义检索双路召回，各取 top-20，合并后做初步去重，目标是**不遗漏任何相关文档**；
2. **精排阶段**：用 Cross-Encoder 对粗筛结果做「问题 - 文档」相关性重打分，排序后取 top-5，目标是**精准筛选出真正能回答问题的优质文档**；
3. **TopK 选型**：最终给大模型的 top-K，用业务验证集在 3/5/7/10 中调优，平衡上下文长度和召回效果。

## 1. 逐环节深度拆解

### 1.1. 粗筛阶段：双路召回，目标「不漏」

##### 核心动作

- **BM25 关键词检索**：基于传统倒排索引，匹配问题和文档的关键词重合度，擅长抓「字面匹配」的内容，比如问题里的专有名词、特定术语；
- **向量语义检索**：用 Embedding 模型把问题和文档转成向量，算余弦相似度，擅长抓「语义相似」的内容，比如同义词、 paraphrase（换说法）的内容；
- **合并策略**：两路各取 top-20（可根据数据量调整），合并后做简单去重，得到一个 40 条左右的候选集。

##### 为什么这么做？

- 单靠 BM25：只会抓字面匹配，语义相关但用词不同的内容会漏；
- 单靠向量检索：容易召回语义相关但不解决问题的「泛内容」，且对关键词不敏感；
- 双路互补：把「关键词命中」和「语义匹配」的内容都捞出来，**最大程度不丢相关文档**，为后续精排留足优质候选。

#### 1.2. 精排阶段：Cross-Encoder 重打分，目标「不错」

##### 核心动作

把问题和每一篇候选文档，以「[CLS] 问题 [SEP] 文档 [SEP]」的格式输入 Cross-Encoder 模型，让模型直接判断「这篇文档能不能回答这个问题」，输出一个 0-1 的相关性分数，按分数从高到低排序，取 top-5。

##### 为什么用 Cross-Encoder？

- 对比粗筛的双路检索：

  - BM25 / 向量检索都是**单塔模型**：问题和文档分开编码，只算「相似度」，不知道文档是否真的能回答问题；
  - Cross-Encoder 是**双塔模型**：问题和文档一起输入，能深度理解「问题 - 文档」的语义关联，精准判断文档的有用性；

- 对比 RRF（ reciprocal rank fusion， reciprocal 排名融合）：

  RRF 只是把两路的排名做加权融合，只看排名、不看内容，**没法判断文档是否真的能回答问题**；而 Cross-Encoder 能做「内容级的相关性判断」，这是 RRF 做不到的，也是精排的核心价值。

##### 为什么取 top-5？

- 平衡「上下文窗口」和「信息质量」：给大模型的文档太多会占满上下文，太少会丢关键信息；top-5 是行业通用的黄金平衡点，既保证信息充足，又不会让模型被冗余内容干扰。

#### 1.3. TopK 参数：用验证集调优

- 核心逻辑：top-K 不是固定值，要根据**业务场景、文档长度、大模型上下文窗口**来定；

- 调优方法：用标注好的验证集，分别测试 top-3/5/7/10 的召回率、准确率、大模型回答效果，选最优值；

- 常见选型：

  - 短文档、小上下文：选 top-3；
  - 通用场景：选 top-5；
  - 长文档、复杂问题：选 top-7/10。

  

## 2. 完整 RAG 检索流程串讲（从用户提问到给大模型）

1. **用户提问**：比如「RAG 两阶段检索是什么？」
2. **粗筛双路召回**：
   - BM25：检索包含「RAG」「两阶段检索」「粗筛」「精排」等关键词的文档，取 top-20；
   - 向量检索：把问题转成向量，找语义最相似的文档，取 top-20；
   - RRF合并去重，得到 40 条候选；
3. **精排重打分**：Cross-Encoder 给 40 条文档打分，排序后取 top-5；
4. **构造 Prompt**：把 top-5 文档拼入上下文，喂给大模型生成回答；
5. **调优迭代**：用验证集调整 top-K、粗筛数量、精排模型，持续优化效果。



## 3. 面试高频追问 & 标准答案

#### 追问 1：为什么不用 RRF 做精排？

**标准答案**：RRF 只是对两路检索的排名做加权融合，**只看排名、不看内容**，它无法判断文档是否真的能回答用户的问题；而 Cross-Encoder 是把问题和文档一起输入，能深度理解两者的语义关联，精准判断文档的有用性，这是 RRF 做不到的，所以必须用 Cross-Encoder 做精排。

#### 追问 2：粗筛为什么用 BM25 + 向量双路，不用单路？

**标准答案**：BM25 擅长关键词匹配，能抓字面相关的内容，但漏语义相关的；向量检索擅长语义匹配，能抓换说法的内容，但容易召回泛内容；双路互补，能最大程度覆盖相关文档，保证「不漏」，为精排提供优质候选。

#### 追问 3：Cross-Encoder 和 Bi-Encoder（向量检索用的 Embedding 模型）的区别？

**标准答案**：

- Bi-Encoder（双塔）：问题和文档分开编码，速度快，适合大规模粗筛，但只能算相似度，无法深度理解问题 - 文档的关联；
- Cross-Encoder（单塔）：问题和文档一起编码，精度高，适合小规模精排，能精准判断相关性，但速度慢，不适合大规模检索。



## 4. RRF

**RRF（Reciprocal Rank Fusion， reciprocal 排名融合）** 是一种**无参数、轻量、鲁棒的多检索结果融合算法**，核心作用是把多个不同检索方式（比如 BM25 关键词检索、向量语义检索）的排名结果，融合成一个统一的最终排名，不用调参就能实现 1+1>2 的效果。

#### 4.1. 应用场景

RRF 是**粗筛阶段双路召回的经典融合方案**，用来把 BM25 和向量检索的结果合并成一个候选集，给后续精排用。

- 步骤 1：BM25 检索取 top-20，向量检索取 top-20
- 步骤 2：用 RRF 融合两路排名，得到一个统一的 top-40（去重后）候选集
- 步骤 3：把候选集丢给 Cross-Encoder 做精排

#### 4.2. 为什么 RRF 适合粗筛？

- 无参数：k 默认 60，几乎不用调，开箱即用
- 鲁棒性强：不受不同检索路数的分数尺度影响（BM25 分数是 0-10，向量分数是 - 1 到 1，RRF 只看排名，不看原始分数）
- 速度极快：纯数学计算，毫秒级完成，不影响检索 latency
- 效果稳定：比简单的「两路取并集」「加权求和」效果好很多

## 5. 核心优缺点

#### 优点

1. **无参数、开箱即用**：k 默认 60，几乎不用调参，适配绝大多数场景
2. **鲁棒性极强**：不依赖检索分数的绝对数值，只看相对排名，不同检索路数（BM25 / 向量 / 混合）都能融合
3. **速度极快**：计算复杂度极低，适合大规模实时检索
4. **效果稳定**：能有效融合不同检索方式的优势，提升召回率和准确率
5. **实现简单**：几行代码就能搞定，工程成本极低

#### 缺点（对应你图里的追问：为什么 RRF 不够？）

1. **只看排名，不看内容**：RRF 只根据文档在各路检索中的排名计算分数，**完全不理解问题和文档的语义关联**，无法判断文档是否真的能回答用户问题
2. **无法做精准相关性判断**：只能做「粗融合」，不能替代 Cross-Encoder 的精排
3. **对长尾文档不友好**：只在少数路排名靠前的文档，很难拿到高总分
4. **无法利用文档特征**：不考虑文档长度、时效性、权重等额外信息（可通过加权 RRF 优化，但会增加复杂度）

## 6. RRF vs Cross-Encoder

|   维度   |            RRF             |          Cross-Encoder          |
| :------: | :------------------------: | :-----------------------------: |
| 核心逻辑 |     基于排名的数学融合     |      基于语义的相关性打分       |
|   输入   |     多路检索的排名结果     |       问题 + 文档的文本对       |
|   速度   |       极快（毫秒级）       |        慢（需模型推理）         |
|   精度   |        低（粗筛用）        |          高（精排用）           |
| 核心能力 |   合并多路结果，保证不漏   | 判断问题 - 文档相关性，保证不错 |
| 适用阶段 |          粗筛阶段          |            精排阶段             |
| 核心局限 | 不理解内容，无法判断有用性 |    速度慢，不适合大规模检索     |
