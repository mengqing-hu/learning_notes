# 1. ReAct 与 CoT 的根本区别 & 选 CoT 的场景

#### 根本区别

- **CoT（思维链）**：只做**推理**，不与外部交互，是**纯内省式思考**。
- **ReAct**：**Reasoning + Acting 交替进行**，一边思考一边调用工具 / 环境，是**闭环交互智能体**。

一句话总结：

**CoT 是 “想”，ReAct 是 “想 + 做”。**

#### 什么时候选 CoT 不用 ReAct

- 任务**不需要外部工具**（纯数学题、逻辑推理、文本生成）
- 追求**低延迟、低成本**，不想多轮调用 LLM
- 环境不可交互、无工具可用
- 任务简单，多步行动反而增加错误概率

------

# 2. ReAct 与 Plan-and-Execute 的区别、优缺点

#### 核心差异

- **ReAct**：**边想边做，小步迭代**

  思考 → 行动 → 观察 → 再思考……

- **Plan-and-Execute**：**先完整规划，再批量执行**

  先生成完整步骤计划，再依次执行，很少回头改计划。

#### 优缺点

##### ReAct

优点：灵活、适应性强、能处理动态环境、可随时修正

缺点：步骤多、易跑偏、易循环、延迟高

##### Plan-and-Execute

优点：步骤清晰、效率高、少冗余调用、可控性强

缺点：对动态环境鲁棒差，计划一旦错全局崩，难以临场调整

#### 适用场景

ReAct → 开放、动态、未知环境（搜索、问答、工具使用）

Plan-and-Execute → 结构化、固定流程任务（代码生成、多步数学、日程规划）

# 3. Reflection 如何与 ReAct 结合？解决什么问题？

#### 结合方式

在 ReAct 每 N 步或失败后，插入一个**反思步骤**：

1. 回顾历史轨迹
2. 判断是否错误 / 循环 / 低效
3. 给出修正策略
4. 重新进入 Reason → Act 循环

常见形式：Self-Refine、Self-Correct、Reflexion、Self-Improve

#### 解决 ReAct 的痛点

- 反复调用同一工具的**无限循环**
- 推理错误、行动无效
- 轨迹过长、上下文混乱
- 策略次优、走弯路
- 无法从失败中学习

一句话：

**Reflection 让 Agent 不仅会 “做事”，还会 “复盘”。**

# 4. ReAct 中如何选工具？粒度与描述怎么设计？

#### 工具选择原则

- 只提供**完成任务必需**的工具，避免冗余
- 工具能力**正交不重叠**
- 工具接口**简单稳定**，返回格式规范
- 避免高风险工具（删库、支付、写入）

#### 工具粒度

- **太细**：LLM 决策负担重，步骤爆炸

- **太粗**：不可控、难纠错、难反思

  最佳实践：中等粒度，单一职责，可组合。

#### 工具描述（Prompt 关键）

- 清晰说明：**功能、入参、出参、适用场景、不适用场景**
- 使用**固定格式**（JSON / 模板）
- 避免模糊自然语言，否则 LLM 会乱调用

# 5. 历史轨迹过长爆 Token？工程策略

1. **滑动窗口**：只保留最近 K 轮思考 - 行动
2. **摘要压缩**：用 LLM 把历史压缩成短总结
3. **结构化记忆**：只存关键状态（目标、进度、结果），丢弃原始对话
4. **分层记忆**：短期记忆（近期轨迹）+ 长期记忆（向量库 / 键值对）
5. **增量更新**：不拼接全部历史，只传变化部分
6. **轨迹剪枝**：删除无效行动、重复步骤、失败尝试

# 6. ReAct 陷入无限循环，如何工程解决？

#### 为什么 ReAct 会死循环？

典型触发条件：

##### 1. 工具反馈不充分

- Observation 太模糊 / 无信息
- 模型无法判断“是否已经成功”

👉 结果：继续调用工具

##### 2. 缺乏终止信号

- Prompt 没有明确：
  - “什么时候该停止”
  - “什么算最终答案”

👉 模型只能继续 Reason → Act → Reason…

##### 3. Reward / 目标模糊

- 没有清晰优化目标
- 模型无法区分“进展 vs 原地打转”

##### 4. 工具调用不可控

- API返回异常 / 空数据
- 或每次返回 slightly different（导致模型以为有新信息）

#### 检测方法

- 统计**同一工具连续调用次数** > 阈值
- 检测**输入参数完全相同**
- 观察**状态 / 结果无变化**
- 轨迹相似度检测

#### 打破循环策略

- 强制**禁止重复调用**同一工具
- 触发 **Reflection** 反思纠错
- 随机或规则化**切换策略**
- 回滚到上一有效状态重新规划
- 直接终止并返回错误

# 7. Few-shot 示例次优会怎样？如何构建高质量示例？

#### 负面影响

- LLM 模仿次优轨迹 → 步骤冗余、效率低
- 错误推理 / 错误调用工具被学到
- 格式混乱 → 解析失败
- 易陷入类似示例的局部最优

#### 系统构建方法

1. 覆盖**常见路径 + 边界案例**
2. 轨迹**最优、简短、格式严格**
3. 统一结构：Thought → Action → Observation
4. 人工 + 自动筛选：用强化学习 / 回测剔除差示例
5. 按任务类型分组示例，动态选择最相关示例

#### 评估方式

- 成功率、步骤数、工具调用次数
- 格式合规率、幻觉率
- 人工 A/B 测试

# 8. ReAct 延迟高、成本高的三种优化

1. **并行化思考与执行**

   可并行的工具调用同时发，减少串行等待

2. **动态停止机制**

   任务完成立即终止，不继续多余思考步骤

3. **小模型做决策，大模型做推理**

   用轻量 LLM 选工具、格式解析；强 LLM 只做复杂推理

4. **缓存复用**

   相同查询 / 工具结果缓存，避免重复计算

5. **提前规划剪枝**

   用 Plan 模式预先减少不必要行动

# 9. ReAct Agent 如何评测？核心指标

#### 核心指标

- **任务成功率**（最关键）
- **平均步骤数 / 工具调用次数**
- **完成延迟、LLM 调用次数**
- **幻觉率、工具调用错误率**
- **格式解析失败率**
- **鲁棒性**（对抗输入、噪声下表现）
- **人类偏好评分**

#### 评测集构建

- 简单 / 中等 / 困难三级任务
- 含干扰工具、模糊查询、多跳需求
- 自动化评测 + 人工校验

# 10. 用户说 “太慢了”，怎么定位瓶颈？

1. 看 LLM **调用次数**是否过多
2. 看工具调用是否**串行、慢、冗余**
3. 看轨迹是否**过长、循环、反复重试**
4. 看 Prompt 是否**过于冗长**
5. 看网络 / API 延迟
6. 看解析逻辑是否复杂（正则 / JSON 解析）
7. 看是否未做缓存、未做窗口裁剪

定位顺序：

**LLM 次数 → 工具延迟 → 轨迹长度 → Prompt 长度**

# 11. 状态管理的好处 & LangGraph 体现

#### 状态管理价值

- 让 Agent **知道自己做到哪了**
- 避免重复思考、重复行动
- 支持中断、恢复、回滚
- 多步骤间共享信息，不依赖自然语言上下文
- 便于监控、调试、反思

#### LangGraph 如何实现

- 显式定义 **State 结构**（节点共享状态）

- 轨迹以**图结构**执行，而非线性字符串

- 支持条件跳转、循环、子图、并行

- 状态可持久化、可观测、可干预

  一句话：LangGraph = 带**状态**的可编程 **ReAct** 执行引擎

# 12. 电商 “自动比价 + 下单”ReAct Agent 安全与鲁棒性

#### 安全性

- **工具权限最小化**：只读查询，下单必须人工二次确认
- 敏感操作（支付、地址）**禁止自动化**
- 输入严格校验、过滤恶意参数
- 日志全记录，可审计可回溯
- 限制调用频率，防爬虫风控

#### 鲁棒性

- 页面结构变化 → 用**稳定 API 优于爬虫**
- 库存 / 价格动态变化 → 每次行动前校验
- 异常处理：售罄、下架、优惠券失效
- 重试策略、超时保护、防循环
- 状态持久化，崩溃可恢复
- 多渠道比价，避免单源数据错误
