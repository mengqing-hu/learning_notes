# 1. 如何设计一个高效的 Agent 上下文维护方案？

核心思路：**分层存储 + 动态压缩 + 结构化管理 + 按需召回**

1. **分层记忆**
   - 短期记忆：最近 N 轮对话 / 轨迹（滑动窗口）
   - 中期记忆：关键状态、目标、进度（结构化字段）
   - 长期记忆：向量库存储历史经验、用户偏好、知识库

2. **动态压缩**
   - 无用信息直接剪枝（重复、失败、无意义步骤）
   - 用 LLM 对长历史做摘要
   - 只保留 Observation 关键结果，去掉冗余思考

3. **结构化上下文**
   - 不用纯自然语言，用 State 结构体：goal、steps、results、errors
   - 减少 LLM 理解成本，降低 token 消耗

4. **检索增强** 
   - 超长历史不全部塞进 prompt
   - 用向量检索召回相关片段，实现 “无限上下文”

5. **增量更新** 
   - 只传变化部分，不传完整历史
   - 支持状态覆盖、合并、回滚

6. **清理机制**
   - 任务结束重置上下文
   - 循环 / 无效步骤自动剔除

# 2. 目前主流 AI 模型上下文长度

- GPT-3.5-turbo：**16k / 128k**
- GPT-4：**8k / 32k / 128k**
- GPT-4o：**128k**
- Claude 3：**200k / 1M**
- Llama 2 / 3：7B/8B **4k/8k**，70B **32k+**
- Qwen、GLM、DeepSeek：**32k / 128k** 常见
- 长文本专用模型可到 **1M+ tokens**

# 3. 完整 Agent 智能体架构

标准模块化架构：

1. **用户接口层**：输入解析、意图识别
2. **记忆模块（Memory）**：短期 / 长期 / 向量记忆
3. **推理模块（Reasoning）**：CoT、ReAct、Plan
4. **规划模块（Planning）**：任务拆解、步骤调度
5. **工具调用模块（Tool Use）**：Tool 注册、参数解析、执行
6. **执行引擎**：状态机、循环控制、异常处理
7. **反思模块（Reflection）**：纠错、优化、自迭代
8. **输出模块**：结果整合、格式化返回

# 4. Agent 和 LLM 的区别

- **LLM**：基座模型，擅长生成、推理、理解，但**不会主动行动、不能调用工具、没有状态**。
- **Agent**：以 LLM 为大脑，**具备自主决策、工具使用、记忆、规划、循环执行**能力的智能系统。

一句话：

**LLM 是大脑，Agent 是带手、脚、记忆的完整智能体。**

# 5. 向量检索 vs 关键词检索

- **关键词检索（BM25/ES）**
  - 基于词频、匹配度
  - 快、轻量、可解释
  - 不理解语义，同义词 / 相关词效果差

- **向量检索（Embedding + 向量库）**
  - 基于语义相似度
  - 能理解意图、上下文、同义表达
  - 适合开放问答、模糊查询
  - 成本更高，依赖 embedding 质量

# 6. Tool 调用机制（标准流程）

1. **Tool 注册**：定义工具名、入参、描述、功能
2. **LLM 决策**：模型根据问题生成 Thought + Action
3. **格式解析**：从输出中提取工具名、参数（JSON / 函数调用格式）
4. **参数校验**：合法性、类型、范围检查
5. **工具执行**：调用函数 / API / 外部系统
6. **Observation 返回**：结果拼回上下文
7. **循环判断**：是否需要继续调用，或直接回答

主流格式：OpenAI **Function Calling**、ReAct **Thought→Action→Obs**

# 7. 如何定位 MySQL 慢查询

1. 开启 **slow_query_log**，设置阈值

2. 用 **mysqldumpslow** 分析日志

3. **explain** 查看执行计划

   - 看 type、key、rows、Extra

4. 检查是否**无索引、索引失效、回表过多**

5. 查看大事务、锁等待、IO 瓶颈

6. 优化 SQL：减少 *、避免子查询、优化 join、合理分页

7. 检查表设计：字段冗余、数据量过大

# 8. MySQL 事务隔离级别

由低到高：

1. **读未提交（Read Uncommitted）**：脏读
2. **读已提交（Read Committed）**：不可重复读
3. **可重复读（Repeatable Read）**：MySQL 默认，解决不可重复读，有间隙锁
4. **串行化（Serializable）**：完全串行，无并发问题，性能差

# 9. 什么是索引覆盖？如何优化？

- **索引覆盖**：查询需要的**所有字段都在索引里**，不需要回聚簇索引查数据。

- 表现：explain 中 Extra 出现 **Using index**

- 优化：

  - 避免 select *
  - 建立联合索引，把查询字段包含进去
  - 减少回表，大幅提升速度

# 10. 进程、线程、协程

- **进程**：操作系统资源分配的最小单位，独立内存空间，开销大
- **线程**：CPU 调度最小单位，共享进程资源，内核调度
- **协程**：用户态轻量级线程，**用户态调度**，切换极快，无内核开销

# 11. 协程 vs 线程

- 协程：用户态切换，极小开销，单线程可跑上万协程，非抢占式

- 线程：内核调度，有上下文切换开销，抢占式，受 OS 管控

  一句话：**协程是轻量级线程，性能更高，但不能利用多核**

# 12. HTTPS 工作原理

1. 客户端发起 HTTPS 请求

2. 服务器返回**证书 + 公钥**

3. 客户端验证证书合法性

4. 客户端生成**随机对称密钥**，用服务器**公钥**加密发送

5. 服务器用**私钥**解密得到**对称密钥**

6. 后续通信**全部用对称密钥加密传输**

   核心：非对称加密握手 + 对称加密传输
