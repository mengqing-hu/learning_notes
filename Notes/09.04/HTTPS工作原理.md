# HTTPS 工作原理

HTTPS = **HTTP + TLS/SSL**，本质是在 HTTP 与 TCP 之间加了一层**加密层**，实现**加密传输、身份认证、数据完整性校验**。

#### 一、核心思想

1. **非对称加密（公钥 + 私钥）**：只用来**安全交换对称密钥**，速度慢但安全。
2. **对称加密**：真正用于传输网页数据，速度快。
3. **数字证书**：防止公钥被掉包，验证服务器身份。

一句话概括：

**先用证书 + 非对称加密安全协商出对称密钥，后续通信全部用对称密钥加密传输。**

#### 二、完整流程（TLS 握手 + 数据传输）

##### 1. 客户端发起 HTTPS 请求

浏览器访问 `https://xxx`，默认端口 **443**。

##### 2. 服务器返回数字证书

证书包含：

- 服务器公钥
- 域名、有效期
- 颁发机构（CA）信息
- 签名（用于防篡改）

##### 3. 客户端验证证书合法性

客户端（浏览器 / 操作系统）内置了受信任的 CA 根证书：

1. 用 CA 公钥验证证书签名是否有效
2. 检查域名、有效期是否合法
3. 证书不合法会直接报 “不安全”

##### 4. 客户端生成随机对称密钥

客户端生成一个**会话密钥（对称密钥）**，

用**服务器公钥加密**后发给服务器。

##### 5. 服务器用私钥解密得到对称密钥

只有服务器持有私钥，能解开密文，得到对称密钥。

至此，双方安全共享同一个密钥，且不会在网络明文传输。

##### 6. 进入加密通信阶段

后续所有 HTTP 请求、响应都**用对称密钥加密**：

- 客户端：明文 → 对称加密 → 密文发送
- 服务器：密文接收 → 对称解密 → 明文

##### 7. 数据完整性校验

传输时附带 **MAC 摘要**，防止数据被篡改。

#### 三、三种关键技术作用

1. **非对称加密**

   公钥加密、私钥解密，安全交换密钥。

2. **对称加密**

   加解密速度极快，适合大量数据传输。

3. **数字证书 + CA**

   防止中间人攻击，保证你连接的是真实服务器。

#### 四、极简流程记忆

1. 客户端请求 HTTPS
2. 服务器返回证书（带公钥）
3. 客户端验证证书
4. 客户端生成对称密钥，用服务器公钥加密发送
5. 服务器用私钥解密得到对称密钥
6. 双方使用对称密钥加密传输数据

------

#### 五、和 HTTP 的区别

- HTTP：明文传输，易窃听、篡改、冒充
- HTTPS：加密传输，安全可靠
- HTTP 端口 80，HTTPS 端口 443
- HTTPS 需要申请证书，略耗性能
